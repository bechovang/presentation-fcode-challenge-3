CÁCH DÙNG REST

Nguyên tắc cốt lõi

1. URL là Danh Từ (Noun)

Không dùng: /getUser
Chỉ dùng: /users

2. Định danh duy nhất

Mỗi dữ liệu có 1 "địa chỉ nhà" riêng.

3. Chuẩn HTTP

Dùng Method (GET, POST) làm hành động.

Ví dụ thực tế

Hành động (Verb) + Địa chỉ (Noun)

GET /users/1 // Đọc User 1
POST /users // Tạo User mới
PUT /users/1 // Sửa User 1
DELETE /users/1 // Xóa User 1
Mẹo: Nhìn URL biết là AI. Nhìn Method biết làm GÌ.

Over-fetching (Lấy thừa)

Tình huống

Chỉ cần "Tên" của User

Hiển thị trên UI

Không cần email, địa chỉ

Minimal data needed

Thực tế REST

GET /users/1 trả về:

{
  "id": 1,
  "name": "Tony",
  "email": "tony@...",
  "address": "...",
  "phone": "...",
  "created_at": "...",
  "updated_at": "..."
}

Wasted bandwidth

HẬU QUẢ: Mua cả nải chuối chỉ để ăn 1 quả. Tốn băng thông 3G/4G, làm chậm Mobile App.

Under-fetching (Lấy thiếu)

Yêu cầu

"Tên User" + "Bài viết" của họ

Dashboard cần cả 2

User profile page

Social media feed

REST phải làm

2 Request riêng biệt:

// Request 1
GET /users/1 → {name: "Tony"}

// Request 2
GET /users/1/posts → [{...}, {...}]

2 round-trips

HẬU QUẢ: Phải gọi 2 lần, mất thời gian chờ đợi (Latency). Mobile mạng chậm → trải nghiệm tệ.
07

N+1 Problem: Cơn ác mộng

Tình huống

Hiển thị News Feed (10 bài)

Yêu cầu: Hiện 10 status + Tên tác giả của từng status.

Vấn đề của dữ liệu:

Trong Database, bảng Post chỉ lưu author_id (số 1, 2...), không lưu Tên.

REST phải làm gì?

GET /posts
-> Trả về 10 bài (chỉ có ID tác giả)

// Bước 2: Phải đi hỏi tên từng người (N request)
Loop (10 lần) {
  GET /users/1 -> "Tên là A"
  GET /users/2 -> "Tên là B"
  ...
}

Giống như đi chợ 10 lần để mua 10 món đồ.

CÔNG THỨC: 1 (Danh sách) + N (Chi tiết) = N+1 Requests
HẬU QUẢ: Danh sách càng dài (N càng lớn) -> App càng lag -> Server càng quá tải.

Client-Driven Queries

Client gửi Query

query {
  user(id: 1) {
    name
    posts {
      title
      createdAt
    }
  }
}

"Cho tôi name và posts"

Server trả về

{
  "data": {
    "user": {
      "name": "Tony",
      "posts": [
        { "title": "Hello", "createdAt": "..." }
      ]
    }
  }
}

ĐÚNG cấu trúc query

TRIẾT LÝ: "Ask for what you need, get exactly that" - Hỏi cái gì, nhận đúng cái đó
10

Fix Over & Under-fetching

Over-fetching → GONE

Chỉ hỏi name

query { user(id: 1) { name } }

→ Chỉ nhận name

Không có email, address

No wasted data

Under-fetching → GONE

Lồng ghép trong 1 query

query {
  user(id: 1) {
    name
    posts { title }
    comments { content }
  }
}

→ 1 request, đầy đủ data

Giống đi siêu thị: Chọn từng món bỏ vào giỏ (Query), thay vì mua combo ép buộc (REST endpoints)

Chiến lược lựa chọn

CHỌN REST KHI

  • Web tin tức, Blog (cần Cache tốt)
  • Hệ thống đơn giản, ít quan hệ
  • Team mới, muốn làm nhanh
  • Public API cho third-party
  • E-commerce catalog
  • KISS principle (Keep It Simple)

CHỌN GraphQL KHI

  • Mobile App (tiết kiệm 3G)
  • Dữ liệu chằng chịt quan hệ
  • Dashboard quản lý phức tạp
  • Frontend thay đổi UI liên tục
  • Microservices aggregation
  • Real-time updates needed
PRO TIP: Frontend cần thay đổi giao diện liên tục mà không muốn nhờ Backend sửa code → GraphQL là vua!
